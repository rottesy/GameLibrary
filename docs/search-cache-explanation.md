# Обоснование пункта: сложный поиск, `JPQL`, `native query`, `Pageable`, in-memory индекс и Postman-сценарий

Ниже подробно объясняется:
1. Что именно реализовано.
2. Почему для фильтрации выбрана вложенная связь `Game -> Achievement`.
3. Чем отличаются `JPQL` и `native query`.
4. Как работает пагинация через `Pageable`.
5. Как устроен in-memory индекс на `HashMap`.
6. Зачем нужен составной ключ и почему важны `equals()` и `hashCode()`.
7. Как реализована инвалидация индекса.
8. Что делает каждый запрос в Postman-коллекции и что он демонстрирует.

## Что именно реализовано

В проект добавлен новый сценарий поиска игр по данным вложенной сущности `Achievement`.

Это покрывает все пункты задания:
1. Есть сложный `GET`-запрос с фильтрацией по вложенной сущности через `@Query (JPQL)`.
2. Есть аналогичный запрос через `native query`.
3. Оба запроса поддерживают `Pageable`.
4. Поверх поисковых запросов добавлен in-memory индекс на `HashMap<CacheKey, Object>`.
5. При изменении данных индекс инвалидируется.

## Почему выбрана связь `Game -> Achievement`

В проекте уже есть готовая вложенная связь:

```java
@OneToMany(mappedBy = "game", fetch = FetchType.LAZY, cascade = CascadeType.ALL, orphanRemoval = true)
private Set<Achievement> achievements = new HashSet<>();
```

И на стороне `Achievement`:

```java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "game_id", nullable = false)
private Game game;
```

Это хороший пример вложенной сущности, потому что:
1. У одной игры может быть много достижений.
2. Фильтрация по полям достижения естественно требует `join`.
3. Можно показать и `JPQL`, и `native SQL` на одной и той же бизнес-задаче.

В нашем случае мы ищем `Game`, но фильтруем результат по полям `Achievement`:
1. `achievementName`
2. `achievementDescription`
3. Дополнительно фильтруем саму игру по `minRating`

То есть запрос не просто ищет запись по одной таблице, а работает через связанную сущность.

## Теория: что такое `JPQL`

`JPQL` — это язык запросов не к таблицам базы данных, а к сущностям JPA.

То есть мы пишем не:

```sql
select * from games g join achievements a on ...
```

а:

```jpql
select g from Game g join g.achievements a
```

Главная идея:
1. Мы работаем с классами `Game`, `Achievement`.
2. Используем связи между сущностями, описанные аннотациями JPA.
3. Hibernate сам преобразует этот запрос в SQL под конкретную БД.

### Плюсы `JPQL`

1. Запрос опирается на модель сущностей, а не на названия таблиц и колонок.
2. Код обычно легче сопровождать при изменениях схемы.
3. Запросы лучше интегрированы в JPA-слой.

### Минусы `JPQL`

1. Меньше контроля над SQL.
2. Иногда трудно использовать специфичные функции конкретной БД.
3. Иногда производительность нужно проверять внимательнее, потому что SQL генерируется Hibernate.

## Теория: что такое `native query`

`native query` — это обычный SQL-запрос, который Spring Data отправляет в БД почти без преобразований.

Пример из проекта:

```sql
select distinct g.*
from games g
join achievements a on a.game_id = g.id
...
```

Здесь мы уже работаем напрямую с таблицами:
1. `games`
2. `achievements`
3. колонками `g.id`, `a.game_id`, `a.name`

### Плюсы `native query`

1. Полный контроль над SQL.
2. Можно использовать возможности конкретной СУБД.
3. Иногда проще оптимизировать вручную.

### Минусы `native query`

1. Запрос жёстко привязан к структуре таблиц.
2. При переименовании таблиц и колонок такой код ломается чаще.
3. Обычно он менее абстрактный и менее переносимый.

## Зачем нужны оба варианта

Это важно для задания, потому что они показывают два разных уровня работы с данными:
1. `JPQL` показывает работу на уровне JPA-модели.
2. `native query` показывает работу на уровне реальной SQL-схемы.

По сути, это две реализации одной и той же бизнес-идеи:
“Найти игры по достижениям”.

## Теория: зачем в кастомных запросах нужен `countQuery`

Когда мы возвращаем `Page<Game>`, Spring Data должен знать:
1. какие элементы попадают на текущую страницу,
2. сколько всего элементов найдено,
3. сколько всего страниц получится.

Для этого недостаточно только основного запроса.
Нужен ещё отдельный запрос подсчёта:

```sql
select count(...)
```

Именно поэтому в кастомных запросах с `Pageable` мы передаём:
1. `value` — основной запрос,
2. `countQuery` — запрос для подсчёта общего количества записей.

Без `countQuery` пагинация либо работала бы некорректно, либо Spring попытался бы сам вывести запрос подсчёта, что на сложных `join`-запросах бывает ненадёжно.

## Теория: как работает `Pageable`

### Что такое пагинация простыми словами

Пагинация — это разбиение большого результата на части, то есть на страницы.

Самая простая аналогия:
1. у тебя есть книга на 300 страниц,
2. ты не читаешь всю книгу сразу,
3. ты открываешь нужную страницу.

С данными на сервере та же идея:
1. есть большой список записей,
2. клиент запрашивает не всё сразу,
3. а только небольшой кусок.

Например:
1. не 10 000 игр сразу,
2. а только 10 или 20 игр на одной странице.

### Как это выглядит в API

Клиент передаёт параметры:
1. `page` — номер страницы,
2. `size` — сколько элементов должно быть на странице,
3. `sort` — в каком порядке их вернуть.

Если взять пример:
1. `page=0`
2. `size=1`

то сервер вернёт только один элемент из общего списка.

Если потом запросить:
1. `page=1`
2. `size=1`

сервер вернёт уже следующий элемент.

`Pageable` — это объект Spring Data, который описывает:
1. номер страницы,
2. размер страницы,
3. сортировку.

Например:
1. `page=0` — первая страница,
2. `size=1` — по 1 элементу на страницу,
3. `sort=title,asc` — сортировка по `title` по возрастанию.

### Почему первая страница — это `0`, а не `1`

В Spring Data нумерация страниц начинается с нуля:
1. `page=0` — первая страница,
2. `page=1` — вторая,
3. `page=2` — третья.

Это связано с тем, что внутри пагинация обычно опирается на смещение от начала выборки.

### Как пагинация связана с `LIMIT` и `OFFSET`

Если говорить языком SQL, пагинация обычно сводится к двум идеям:
1. сколько строк взять,
2. сколько строк пропустить перед этим.

В SQL это обычно выражается так:

```sql
LIMIT size
OFFSET page * size
```

Пример:
1. `page=0`, `size=10` -> `OFFSET 0`
2. `page=1`, `size=10` -> `OFFSET 10`
3. `page=2`, `size=10` -> `OFFSET 20`

То есть каждая следующая страница сдвигает “окно просмотра” дальше по результатам запроса.

### Что возвращается наружу

В контроллере мы возвращаем `Page<GameResponse>`.
Это не просто список.
Это структура, которая содержит:
1. `content` — сами данные,
2. `number` — номер текущей страницы,
3. `size` — размер страницы,
4. `totalElements` — сколько найдено всего,
5. `totalPages` — сколько страниц всего.

Это очень важно, потому что клиенту мало просто получить кусок данных.
Ему ещё нужно понимать:
1. сколько данных найдено всего,
2. есть ли следующая страница,
3. на какой странице он сейчас находится.

### Пример пагинации на нашем сценарии

Предположим, что запрос нашёл 2 игры и мы используем:
1. `size=1`
2. `sort=title,asc`

Тогда:
1. `page=0` вернёт первую игру,
2. `page=1` вернёт вторую игру,
3. `totalElements = 2`,
4. `totalPages = 2`.

Именно для такой наглядности в коллекции создаются две игры.

### Почему пустая страница — это не ошибка

Если записей мало, а номер страницы большой, страница может быть пустой.

Например:
1. найдено всего 2 записи,
2. `size=1`,
3. тогда есть только страницы `0` и `1`,
4. а `page=2` уже вернёт пустой `content`.

Это нормальное поведение.
Пустая страница означает не сбой сервера, а то, что данных для такого диапазона уже нет.

### Почему пагинация важна

Если данных много, то отдавать их все одним ответом плохо:
1. растёт время ответа,
2. увеличивается нагрузка на БД и сеть,
3. клиенту неудобно работать с огромным массивом.

`Pageable` решает эту проблему и делает запрос пригодным для реальных сценариев.

### Практический смысл пагинации

Пагинация нужна не “для галочки”, а для реальной управляемости системы:
1. API не отдаёт слишком большие ответы,
2. интерфейс может подгружать данные постепенно,
3. база меньше нагружается на больших выборках,
4. пользователь быстрее получает первую часть результата.

Именно поэтому во многих системах пагинация используется почти в каждом поиске и списочном API.

## Теория: что такое in-memory индекс в этом задании

Под “in-memory индексом” здесь фактически понимается прикладной кеш ранее выполненных запросов.

Идея такая:
1. Пользователь отправляет поисковый запрос.
2. Мы формируем ключ из параметров запроса.
3. Если такой ключ уже есть в `HashMap`, возвращаем готовый результат из памяти.
4. Если ключа нет, идём в БД, получаем результат и сохраняем его в карту.

Это уменьшает количество одинаковых обращений к базе для повторяющихся запросов.

## Почему выбран `HashMap`

`HashMap` подходит здесь лучше всего, потому что:
1. она хранит пары `ключ -> значение`,
2. среднее время поиска по ключу близко к `O(1)`,
3. очень удобно реализовать шаблон “найти или вычислить”.

Именно это делает метод:

```java
computeIfAbsent(...)
```

Логика у нас такая:
1. Есть ключ — вернули кеш.
2. Нет ключа — вызвали supplier, получили данные, сохранили в карту.

## Почему здесь нужен составной ключ

Если бы ключом был только метод, например:

```java
"findByAchievementsWithJpql"
```

то все запросы этого метода конфликтовали бы между собой.

Например, эти два запроса нельзя считать одинаковыми:
1. `achievementName=Master Explorer&page=0&size=1`
2. `achievementName=Master Explorer&page=1&size=1`

Также отличаются:
1. `JPQL` и `native` версии,
2. разные `minRating`,
3. разная сортировка,
4. разное описание достижения.

Поэтому ключ должен учитывать все параметры, влияющие на результат.

В проекте в ключ включены:
1. класс сущности,
2. имя метода,
3. `achievementName`,
4. `achievementDescription`,
5. `minRating`,
6. номер страницы,
7. размер страницы,
8. сортировка.

## Почему важны `equals()` и `hashCode()`

`HashMap` работает корректно только если ключ умеет правильно:
1. сравниваться на равенство (`equals()`),
2. вычислять хэш (`hashCode()`).

Если этого нет или реализация неправильная:
1. одинаковые по смыслу ключи будут считаться разными,
2. карта не будет находить уже закешированные значения,
3. кеш станет бесполезным или начнёт вести себя непредсказуемо.

### Как это решено в проекте

`CacheKey` сделан через `record`:

```java
public record CacheKey(Class<?> entityClass, String methodName, List<Object> args)
```

У `record` `equals()` и `hashCode()` генерируются автоматически на основе всех компонентов.

Это удобно, потому что:
1. ключ компактный,
2. сравнение происходит по значениям,
3. код короче и менее подвержен ошибкам.

### Дополнительная важная деталь

В `CacheKey` добавлена защита аргументов:
1. список аргументов копируется,
2. делается неизменяемым,
3. корректно обрабатывает `null`.

Это важно, потому что:
1. ключ после создания не должен “меняться”,
2. иначе у `HashMap` может нарушиться логика поиска,
3. поисковые параметры у нас опциональные, значит `null` или пустое значение нужно обрабатывать безопасно.

## Как работает инвалидация индекса

Проблема любого кеша в том, что данные в памяти могут устареть.

Например:
1. мы один раз выполнили поиск по достижениям,
2. результат сохранился в `HashMap`,
3. потом игра была обновлена или удалена,
4. если кеш не сбросить, мы продолжим отдавать старый результат.

Это называется “грязный кеш” или “устаревшие данные”.

### Что делаем в проекте

После изменения игры вызывается:

```java
cacheManager.invalidate(Game.class)
```

То есть из карты удаляются все записи, связанные с поиском по `Game`.

### Где вызывается инвалидация

В `GameServiceImpl` инвалидация выполняется после:
1. `create(...)`
2. `update(...)`
3. `delete(...)`
4. `createGameWithReviewAndAchievementInternal(...)`

Последний пункт особенно важен, потому что при создании достижения меняется как раз та связанная сущность, по которой мы фильтруем.

## Почему мы инвалидируем по классу, а не по одному ключу

Это простой и надёжный подход.

Когда данные изменились, заранее трудно знать, какие именно комбинации фильтров уже были закешированы:
1. поиск по `achievementName`,
2. поиск по `achievementDescription`,
3. разные страницы,
4. разные сортировки.

Поэтому безопаснее удалить все кеш-записи для `Game.class`, чем пытаться выбирать вручную только часть ключей и случайно оставить устаревшие данные.

Это компромисс:
1. он менее “тонкий”,
2. зато простой,
3. надёжный,
4. полностью подходит для учебного задания.

## Как устроен Postman-сценарий

Коллекция лежит здесь:

`postman/GameLibrary-Search-And-Cache.postman_collection.json`

Она разбита на 3 группы:
1. `Setup Data`
2. `Search With Pagination`
3. `Cache Invalidation Check`

Также используются переменные коллекции:
1. `baseUrl`
2. `developerId`
3. `userId`
4. `gameId`
5. `secondGameId`
6. `achievementName`
7. `achievementDescription`

Часть переменных (`developerId`, `userId`, `gameId`, `secondGameId`) сохраняется автоматически из ответов через встроенные Postman test scripts.

Для наглядной демонстрации пагинации в `Setup Data` создаются две разные игры с одинаковыми значениями:
1. `achievementName`
2. `achievementDescription`

Благодаря этому поисковый запрос находит минимум 2 записи.
А поскольку в поиске используется `size=1`, то:
1. `page=0` содержит первую игру,
2. `page=1` содержит вторую игру.

## Полный разбор всех запросов Postman

### 1. `Create Developer`

**Метод:** `POST`  
**URL:** `/api/developers`

### Что делает

Создаёт разработчика, который нужен для создания игры.

### Почему без него нельзя

В `GameRequest` поле `developerId` обязательное:

```java
@NotNull
private Long developerId;
```

Поэтому перед созданием игры в базе уже должен существовать разработчик.

### Что важно в ответе

Из ответа берётся `id`, который сохраняется в переменную коллекции:

```javascript
pm.collectionVariables.set('developerId', body.id);
```

Потом этот `developerId` используется в следующих запросах.

## 2. `Create User`

**Метод:** `POST`  
**URL:** `/api/users`

### Что делает

Создаёт пользователя, который нужен для создания `Review`.

### Почему он нужен

В `ReviewRequest` есть обязательное поле:

```java
@NotNull
private Long userId;
```

Значит без пользователя мы не сможем создать отзыв в composite-запросе.

### Что важно в ответе

Из ответа сохраняется `userId`.

## 3. `Create First Game With Review And Achievement`

**Метод:** `POST`  
**URL:** `/api/games/with-review-and-achievement/tx`

### Что делает

Это ключевой подготовительный запрос.
Он создаёт сразу:
1. игру,
2. отзыв,
3. достижение.

### Почему он важен

Все дальнейшие поисковые запросы строятся именно вокруг этой игры и её достижения.

### Что здесь демонстрируется

1. создание данных для поиска,
2. работа вложенной связи `Game -> Achievement`,
3. создание результата, который потом ищется и кешируется,
4. транзакционный сценарий сохранения.

### Почему в `review.gameId` и `achievement.gameId` стоит `0`

В коллекции это просто техническая заглушка, потому что поля помечены как обязательные (`@NotNull`), но в сервисе они потом перезаписываются:

1. после сохранения игры берётся `savedGame`,
2. для `Review` вызывается `review.setGame(savedGame)`,
3. для `Achievement` вызывается `achievement.setGame(savedGame)`.

То есть фактически игра для отзыва и достижения определяется сервером, а не значением `0` в JSON.

### Что важно в ответе

Из ответа сохраняется `gameId`.

Он нужен для:
1. `Update Game`
2. `Delete Game`

## 4. `Create Second Game With Review And Achievement`

**Метод:** `POST`  
**URL:** `/api/games/with-review-and-achievement/tx`

### Что делает

Создаёт вторую игру с тем же `achievementName` и `achievementDescription`, что и первая.

### Зачем он нужен

Этот запрос добавлен специально для демонстрации пагинации.

Если бы в выдаче была только одна игра, то:
1. `page=0` содержала бы запись,
2. `page=1` была бы пустой.

Это не ошибка, но для защиты не очень наглядно.

Поэтому мы создаём вторую подходящую игру, чтобы:
1. `page=0` была непустой,
2. `page=1` тоже была непустой,
3. отличие страниц было видно сразу по содержимому ответа.

### Что важно в ответе

Из ответа сохраняется `secondGameId`.

Он не обязателен для дальнейших шагов, но полезен как служебная переменная, если понадобится проверить вторую игру отдельно.

### Что делает

Выполняет тот же поиск, но уже для второй страницы:

Он показывает, что:

1. `page=0`

## 8. `Warm JPQL Cache`

**Метод:** `GET`  
**URL:** `/api/games/search/by-achievements/jpql`

### Что делает

Запускает `JPQL`-поиск один раз, чтобы положить результат в in-memory индекс.

### Зачем нужен

Это подготовка к демонстрации кеширования.

Логика такая:
1. сначала результата в кеше нет,
2. сервис обращается к репозиторию и в БД,
3. затем сохраняет результат в `HashMap`,
4. повторный запрос с теми же параметрами может быть отдан уже из памяти.

### Что именно кешируется

Кешируется не просто “все игры”, а результат конкретного поискового запроса:
1. с конкретным `achievementName`,
2. с конкретным `achievementDescription`,
3. с конкретным `minRating`,
4. с конкретной страницей,
5. с конкретной сортировкой.

## 9. `Update Game`

**Метод:** `PUT`  
**URL:** `/api/games/{{gameId}}`

### Что делает

Изменяет ранее созданную игру.

### Что демонстрирует

Главная цель этого запроса не просто изменить игру, а спровоцировать инвалидацию кеша.

После `save(...)` вызывается:

```java
invalidateGameSearchCache();
```

А внутри:

```java
cacheManager.invalidate(Game.class);
```

### Почему это важно

Если после обновления не сбросить кеш, то последующий поиск может вернуть старую версию игры.

## 10. `JPQL Search After Update`

**Метод:** `GET`  
**URL:** `/api/games/search/by-achievements/jpql`

### Что делает

Повторяет тот же поиск после изменения игры.

### Что демонстрирует

1. что старый кеш уже не используется,
2. что сервис снова идёт за данными,
3. что после обновления возвращается актуальный результат.

### Как это объяснить на защите

До обновления запрос был прогрет в кеше.  
После обновления кеш сброшен.  
Значит этот запрос не должен брать старое значение из `HashMap`.

## 11. `Delete Game`

**Метод:** `DELETE`  
**URL:** `/api/games/{{gameId}}`

### Что делает

Удаляет созданную игру.

### Что демонстрирует

1. изменение данных в системе,
2. повторную инвалидацию кеша,
3. влияние удаления на результаты поиска.

После удаления игра больше не должна появляться в поисковой выдаче.

## 12. `JPQL Search After Delete`

**Метод:** `GET`  
**URL:** `/api/games/search/by-achievements/jpql`

### Что делает

Повторяет тот же самый поиск после удаления игры.

### Что демонстрирует

1. что кеш был сброшен,
2. что результат действительно изменился,
3. что удалённая игра больше не возвращается.

Это финальная проверка корректной инвалидации.

## Как объяснить этот сценарий целиком на защите

Можно рассказывать так:

1. Мы создаём связанные данные: разработчика, пользователя и две игры с одинаковым достижением для поискового сценария.
2. Затем ищем игры по вложенной сущности `Achievement`.
3. Один и тот же поиск реализован двумя способами:
   `JPQL` и `native query`.
4. Оба варианта поддерживают `Pageable`.
5. Повторные одинаковые запросы кешируются в памяти через `HashMap`.
6. Ключ кеша составной и включает все значимые параметры запроса.
7. При изменении или удалении игры кеш инвалидируется, чтобы не отдавать устаревшие данные.

Это полностью покрывает все требования задания.

## Что важно помнить при демонстрации

1. В сущностях `Developer`, `User` и `Game` есть уникальные поля, поэтому в коллекции использованы Postman dynamic variables для повторного запуска без конфликтов по уникальности.
2. `page=0` и `page=1` — это разные ключи кеша.
3. `JPQL` и `native` — тоже разные ключи кеша, потому что это разные методы.
4. Пустые строковые фильтры специально нормализуются, чтобы запросы не падали и одинаково обрабатывались в кеше.
5. `size=1` выбран специально для защиты, чтобы разницу между страницами было видно прямо в `content`.
